# Word Mage（言葉の魔法使い）

## コンセプト
Slay the Spire的ノード進行 × 自然言語書き換え × 手札管理パズルゲーム。
プレイヤーは「校正者（リヴァイザー）」として、文章のスロット（主語・述語・修飾語等）を手札の語と入れ替え、世界を書き換えながら進む。

## 技術スタック
- **フレームワーク**: Svelte 5 + TypeScript
- **ビルド**: Vite（base: '/word-mage/'）
- **データ**: 静的TypeScript（ノード定義、語彙、タグはnodes.ts内）
- **描画**: HTML/CSS（古い本の見開きUI）
- **デプロイ**: GitHub Pages（gh-pagesブランチ、`npx gh-pages -d dist`で更新）
- **将来**: Tauri でデスクトップ/Steam対応

## デプロイ

npm run build && npx gh-pages -d dist

## ゲーム設計

### ノードタイプ
| タイプ | 構造 | 説明 |
|--------|------|------|
| puzzle | 1文 | 環境/危機を書き換えて突破 |
| elite | 1文 | 高難度・高報酬 |
| rest | 1文 | 泉。書き換えなければHP回復、書き換えると面白いイベント |
| battle/boss | 2文・複数ターン | 敵行動文+プレイヤー行動文。敵HPあり |
| shop | 専用UI | 語カード購入・手札売却・アイテム売却 |
| treasure | 専用UI | カードプールからランダムで語カード獲得 |

### 1文構成ノード（puzzle/elite/rest）
- スロット最大4（読みやすさのため）
- 助詞はスロットカテゴリから自動生成（subject→が、object→を、object_ni→に等）
- 判定: Outcome配列を先頭から走査、最初にマッチした結果を適用
- マッチしない場合 → defaultOutcome（文の歪み: 高ダメージ、ゴールド0）
- 「書き換えなかった場合」も明示的にOutcomeとして定義

### 2文構成バトル（battle/boss）
- 敵行動文: 毎ターン新しい攻撃パターン（ローテーション）
- プレイヤー行動文: 毎ターンリセット。手札で強化して敵にダメージ
- APは両文で共有 → 攻防のAP配分がStSの攻撃/ブロック振り分けに相当
- 敵HPが0になれば勝利

### 判定方式: 組み合わせ定義ベース
- 各Outcomeに`conditions`（スロットID→必要タグのAND条件）
- 配置順: 複合条件（具体的）→ 単体条件 → 元テキスト条件
- defaultOutcome = 「文の歪み」（未定義の組み合わせ = 文法破綻とみなす）
- LLMで組み合わせパターンとテキストを大量生成 → 人間がキュレーション

### リソース
- **HP 20/20**: ダメージ/回復。0でゲームオーバー
- **ゴールド**: 戦闘報酬で獲得、ショップで消費
- **手札**: 初期5枚、上限7（永続カードで+2可）。同カテゴリのスロットにのみ入る
- **アイテム**: 換金アイテム（ショップで売れる）。うまい書き換えで入手
- **永続カード**: 手札枠を使ってパッシブ効果（AP+1、手札上限+2、毎ノード+2G）

### マップ
- 5層分岐マップ: puzzle → puzzle/treasure → puzzle/rest/elite → shop → boss
- ノードホバーで右ページにプレビュー（タイプ、名前、スロット構成、AP）
- ノードタイプ別アイコン・色

### 画面設計
- 全画面が「古い童話の本の見開き」
- 左ページ: イベント文 / バトル（敵文+自文）/ マップ / ショップ商品
- 右ページ: 結果テキスト / プレビュー / 売却UI
- 本の下（机上）: 手札エリア
- 上部: HPバー + ゴールド表示（バトル時はプレイヤーセクション内に移動）

### 質感・演出
- ブラウザ臭を排除（テキスト選択無効、右クリック無効）
- 紙のクリーム色 + セピアのグラデーション
- ドラッグ中: 全スロットが微かに光る（drag-ready）、ドロップ先は1.1倍拡大+金色光
- 助詞の自動表示で自然な日本語
- 文の歪み = defaultOutcomeで高ダメージ処理

### 世界観・トーン
- 「書かれた世界」：世界は一冊の書物として存在
- 文体: 乾いた寓話体、三人称現在形
- 物語性はフレーバー程度。「魔法＝言葉のハック」という哲学が通底
- 童話のような古い物語の質感

## 拡張カードロスター（2026-04-02 洗い出し、未確定）

### 選定基準
1. **方向性テスト**: 「Xが道を塞いでいる」で効果の方向がわかるか
2. **ユーモアテスト**: 「屈強なX」で笑えるか。修飾語との化学反応
3. **述語レンジテスト**: 塞ぐ/守る/逃げる/砕くで全部違う絵が浮かぶか
4. 存在そのものが効果の方向性を帯びているか（修飾語・述語に依存しない固有の方向）

### 修飾語（連体修飾語）候補 11
小さな / 優しい / 燃え盛る / 堅い / 鋭い / 壊れた / 屈強な / 凍てついた / 透き通った / 呪われた / 眠い

- 大きな → 屈強な に差し替え（ユーモア優先。サイズ増大より筋肉化のほうが化学反応が起きる）
- 眠い: 壊れた(機能停止/悲しい)と差別化。一時停止/ユーモラス。「眠い竜が道を塞いでいる→起こさなければ通れる」

### 主語候補 17
| カテゴリ | 候補 | 方向性 |
|---------|------|--------|
| 小動物(2) | 子猫, 蛇 | 無害化 / 毒・狡猾 |
| 幻獣(2) | 竜, 蝶 | 圧倒 / 儚さ・変容 |
| 自然力(3) | 風, 炎, 氷 | 吹散 / 焼却・温め / 凍結・保存 |
| 抽象(3) | 影, 歌, 沈黙 | 隠蔽 / 魅了・共鳴 / 抑圧・無音化 |
| 霊(1) | 亡霊 | 畏怖・すり抜け・冷気 |
| 人型(3) | 騎士, 魔女, 王 | 戦闘・防御 / 呪い・変容 / 命令・支配 |
| 物体(2) | 鏡, 人形 | 反射・反転 / 不気味・操り・身代わり |
| 自然現象(1) | 雨 | 水害・浄化・湿り |

存在数: 11修飾語 × 17主語 = 187存在

### 落選とその理由
- 夢: 方向性テスト不合格。修飾語依存の空の器
- 花: 修飾語との化学反応が薄い。屈強な花が面白くない
- 老人: 方向性弱い（老人がいると何が起きる？が不明確）
- 鬼/盗賊: 述語レンジ狭い。どの述語でも「力で」「盗んで」の一本調子
- 狼: 竜と被る（強い獣）。ユーモアテストも弱い
- 大きな: ユーモア弱い。サイズ変更は直感的すぎて意外性がない

### 後から追加する場合のコスト
- 修飾語1枚追加: 存在シート17枚 + 気になるオーバーライド数個
- 主語1枚追加: 存在シート11枚 + 気になるオーバーライド数個
- 初期リリース時に完璧でなくてよい。遊んでから足す

## Outcome設計

### 現行方式（Web簡易版: 2026-04-02実装済み）
- 全有効組み合わせに固有テキストを1つずつ記述（カードID直接指定）
- conditions: `{ s1: 'h1', s2: 'shop_3', s4: 'shop_4' }` （string=カードID, string[]=タグ互換）
- を格で自動詞（消え去る/輝く）→ 歪み。に格/から格は自動詞OK。逃げていくは経路のを格として許容
- 歪みテキストは世界観内で書く（メタ的文法解説はNG）
- 全6通常ノード+ボス4判定文 = 1,156 outcomes 完了

### スケーリング方針（ノード拡張フェーズ用、2026-04-02設計決定）

#### 核心思想: 「存在」が面白さの中心
- **修飾語×主語 = 「存在」**（例: 壊れた子猫, 燃え盛る竜）がこのゲームの楽しさの核
- 存在の「創造的解釈」（壊れた子猫→ぬいぐるみ）が全ノードで再利用される
- ワード・ノードのバリエーションを増やすことがゲームの価値を高める

#### 不変スロットによるノード個性化
- **全スロットを自由に変えられる必要はない**
- ノードごとに「不変要素」を設定し、変更可能なスロットを制限する
- 不変要素がノードの個性・パズルの制約を生む

例:
```
ノードA: [存在]が ゆっくり 川を 渡る     ← 副詞・述語固定、存在のみ可変
ノードB: 恐ろしい [主語]が 旅人を [述語]   ← 修飾語固定、主語+述語可変
ノードC: [存在]が 沼に 逃げていく         ← 述語固定、存在のみ可変
```

| パターン | 可変スロット | 組み合わせ数(30カード時) |
|---------|------------|----------------------|
| 存在のみ | mod+subj | ~96 |
| 存在+述語 | mod+subj+pred | ~576 |
| 存在+副詞 | mod+subj+adv | ~384 |
| 主語+述語（修飾語固定）| subj+pred | ~48 |

#### 存在シート + 状況テンプレート方式
1. **存在シート**（mod×subj = 1枚）: 名前、創造的解釈、特性を定義
   - 例: 壊れた子猫 → 解釈「ぬいぐるみ。綿が出ている。」特性{threat:0, inanimate}
   - 必要数: mod数×subj数（12×8=96枚）。ここが創造性の集中投資先
2. **状況テンプレート**（ノード×述語×特性分岐）: 存在の特性に応じた結果を定義
   - 例: node_1+block+threat:0 → "{name}が道を塞いでいる。{interpretation}脅威はない。"
   - 必要数: 50ノード × 4-6述語 × 5-10分岐 ≒ 1,000〜3,000
3. **手書きオーバーライド**: 特に面白い組み合わせだけ完全手書き
   - 必要数: 500〜1,000

#### スケーリング見積もり
| 規模 | 存在シート | テンプレート | オーバーライド | 合計 |
|------|-----------|------------|--------------|------|
| 現在(15カード/10判定文) | 28 | - | 1,156(全手書き) | ~1,200 |
| 中規模(30カード/50ノード) | 96 | ~2,000 | ~700 | ~2,800 |
| 大規模(50カード/100ノード) | ~200 | ~4,000 | ~1,000 | ~5,200 |

#### 新コンテンツ追加コスト
| 追加内容 | 作業量 |
|---------|--------|
| 修飾語1枚 | 存在シートに主語数ぶん追加 + 気になるオーバーライド数個 |
| 主語1枚 | 存在シートに修飾語数ぶん追加 + 気になるオーバーライド数個 |
| 述語1枚 | 述語可変ノードのテンプレートに分岐追加。述語固定ノードは影響なし |
| ノード1個 | 不変要素を決めて状況テンプレート作成。存在シート参照で自動対応 |

#### 移行計画
- Web簡易版（現在）: 全手書き方式。1,156 outcomesの手書きデータが存在シート設計の資料になる
- ノード拡張フェーズ: 存在シート+状況テンプレート+オーバーライド方式に移行
- LLMワークフロー: Sonnet下書き→Opus磨き上げ が効率的と確認済み

## 通貨: Quill (Q)（2026-04-02決定、未実装）

- gold（ゴールド）→ Quill（Q）にリネーム予定
- 概念: 書き換えの腕前が筆に宿る力。うまい書き換えほどQが多い
- UI表示: 「Q」を大文字で大きめ、「uill」を右に小さく。略称Q
- ショップ: Qで新しい語カードを獲得（筆の力で語を呼ぶ）
- アイテム→Q変換: 品を筆に吸わせて力に変える
- 付与基準（要精緻化）: damage低→Q多、damage高→Q少、歪み→Q 0
